# OTFIT GDPR 적용 리포트

> 작성: 2026-07-22 · 팀 Boolean (CODEGATE 2026 AI 스타트업 해커톤)
> 범위: 백엔드(main 브랜치) 구현 기준. 각 항목은 코드 위치와 자동화 테스트로 검증됨.

OTFIT은 사용자의 **얼굴이 포함된 사진**을 처리하는 서비스로, 개인정보 보호를 기능이 아닌
설계 전제로 삼았다. 본 문서는 GDPR(일반 개인정보 보호법)의 요구사항이 코드의 어느 지점에
어떻게 적용되어 있는지 정리한다.

## 1. 처리하는 개인정보 인벤토리

| 데이터 | 수집 시점 | 보관 | 비고 |
|---|---|---|---|
| 이메일·닉네임·(비밀번호 해시) | 가입 | 탈퇴 시까지 | 소셜 가입은 비밀번호 없음 |
| 소셜 식별자 (provider, provider_id) | 소셜 로그인 | 탈퇴 시까지 | 소셜 토큰은 검증 후 폐기(미저장) |
| 사용자 사진 (원본) | 업로드 | **30일 자동 파기** | EXIF 제거 후 저장 |
| 생성 결과 이미지 | AI 피팅 | 탈퇴 시까지 | 게시 여부는 사용자 선택 |
| 게시물·투표·댓글 | SNS 활동 | 탈퇴 시까지 | |
| 행동 이벤트 (퍼널) | 사용 중 | 통계 목적 | 상품 클릭 등, 회원 식별 기반 |

## 2. 조항별 적용 현황

### ✅ 적용 완료 (코드 + 테스트)

| 조항 | 요구사항 | 구현 위치 | 검증 |
|---|---|---|---|
| **Art. 6·7** 적법 근거·동의 | 처리 전 명시적 동의, 동의 기록, 철회 가능 | 업로드 시 `consent_image_processing` 미동의 → 422 거부 (`app/services/photos.py: upload`), 동의 이력 테이블 (`app/models/consent.py`), 철회 `POST /consents granted=false` | `test_e2e_flow`, 업로드 경로 테스트 |
| **Art. 5(1)(c)** 데이터 최소화 | 필요한 최소만 수집 | 가입 3항목(이메일·비번·닉네임)만, 사진 저장 전 **재인코딩으로 EXIF(GPS·기기정보) 제거** (`app/services/photos.py`) | `test_security.py::test_upload_strips_exif_and_limits_size` |
| **Art. 5(1)(e)** 보관 제한 | 목적 달성 후 파기 | 사진 `delete_after` = 업로드+30일, Celery 주기 작업이 자동 삭제 (`app/workers/celery_app.py`, `app/repositories/photos.py`) | 만료 사진 조회 쿼리 |
| **Art. 15·20** 접근권·이동권 | 보유 데이터 사본을 구조화된 형식으로 | `GET /me/export` — 프로필·동의·사진 메타·생성·게시물·투표·댓글·크레딧·팔로잉을 JSON으로 (`app/services/privacy.py`) | `test_privacy.py` |
| **Art. 17** 삭제권 | 잊혀질 권리, 지체 없는 삭제 | ① 사진 즉시 삭제 `DELETE /photos/{id}` (스토리지 파일 포함) ② **계정 완전 삭제 `DELETE /me`** — 원본·생성결과 파일 삭제 + FK CASCADE로 게시물·투표·댓글·크레딧 일괄 소멸 (`app/services/privacy.py: delete_account`) | `test_privacy.py` — 삭제 후 토큰 무효·재로그인 불가·피드 소멸 검증 |
| **Art. 25** 설계·기본값에 의한 보호 | privacy by design/default | 비포(원본) 사진 **기본 비공개**, 사용자가 명시(`include_before`)할 때만 공개 (`app/services/posts.py`) · 모든 리소스 소유권 검증(본인 사진·결과만 접근) | `test_sns.py`, 403 경로 |
| **Art. 32** 처리 보안 | 적절한 기술적 보호조치 | bcrypt 해시, JWT access/refresh 분리, 소셜 토큰 서버 재검증(+구글 aud), 인증 레이트리밋(분당 20), 보안 응답 헤더, 업로드 10MB 제한, SECRET_KEY 기본값 경고 (`app/core/hardening.py`, `app/core/security.py`) | `test_security.py` 3종 |

### ⚠️ 부분 적용 / 로드맵 (정식 출시 전 필요)

| 조항 | 상태 | 계획 |
|---|---|---|
| **Art. 13·14** 정보 고지 | 동의 체크는 있으나 개인정보처리방침 문서 미게시 | 앱 온보딩·설정에 처리방침 페이지 추가 (처리 목적·보관기간·제3자 제공 명시) |
| **Art. 28 / 44~** 처리자·제3국 이전 | 사용자 사진이 생성 시 **외부 AI(Segmind·OpenAI, 미국)로 전송**됨. presigned URL은 1시간 만료로 노출 최소화 | 처리자 계약(DPA) + 표준계약조항(SCC) 체결, 또는 EU 리전/온프레미스 생성으로 전환. 동의 문구에 제3자 AI 처리 명시 |
| **Art. 9** 특수범주 (생체정보) | 사진 자체는 특수범주 아님. 단 live 품질검사의 얼굴 유사도 비교(`face_similarity`)는 해석 여지 있음 | "신원 식별이 아닌 품질 검증 목적"임을 방침에 명시, 필요 시 별도 동의 항목 분리 |
| **Art. 33·34** 유출 통지 | 프로세스 문서 없음 | 72시간 통지 절차를 운영 문서로 수립 |
| **Art. 8** 아동 동의 | 연령 확인 없음 | 가입 시 만 14세(한국 기준)/16세(EU) 확인 추가 |

## 3. 데이터 흐름 요약 (사진 기준)

```
업로드 → [동의 강제 + 10MB 제한 + EXIF 제거 + JPEG 재인코딩] → 스토리지
       → 분석·생성 (live 시 presigned URL 1시간 → 외부 AI)   ← 로드맵: DPA/SCC
       → 결과 게시 여부는 사용자 선택 (원본은 기본 비공개)
       → 30일 자동 파기 / 사용자 즉시 삭제 / 탈퇴 시 전체 파기
```

## 4. 결론

데이터 주체의 6대 권리 축(동의·최소화·보관 제한·접근·이동·삭제)은 **전부 실제 API와
자동화 테스트로 구현**되어 있으며, 남은 항목은 문서·계약 성격(처리방침 게시, DPA/SCC)이다.
따라서 현 시점 상태는 "GDPR 완전 준수"가 아닌 **"GDPR-ready 설계"**로 표현하는 것이 정확하다.

관련 국내법(개인정보보호법) 관점에서도 동일 구조가 적용된다 — 동의(제15조), 파기(제21조),
열람·전송(제35조), 삭제(제36조), 안전성 확보조치(제29조).
